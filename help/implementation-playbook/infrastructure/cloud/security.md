# 安全性

Adobe Commerce Pro計畫體系結構旨在提供高度安全的環境。 每個客戶都部署到與其他客戶分開的各自獨立伺服器環境中。 以下說明生產環境的安全性詳細資訊。

## 網頁瀏覽器

進入和離開雲端環境的大部分流量來自消費者的網頁瀏覽器。 針對網站上的所有頁面，消費者流量可使用HTTPS加以保護（使用共用SSL憑證或客戶自己的SSL憑證，但需支付額外費用）。 結帳和帳戶頁面一律使用HTTPS提供。 最佳實務是以HTTPS提供所有頁面。

## 內容傳遞網路(CDN)

以快速方式提供CDN和分佈式拒絕服務(DDoS)保護。 Embly CDN有助於隔離直接存取來源伺服器。 公共DNS只指向「快速網路」。 Emblyt DDoS解決方案可抵御高破壞性的第3層和第4層攻擊，以及更複雜的第7層攻擊。 可使用根據整個HTTP/HTTPS要求，並根據用戶端和要求條件（包括標題、Cookie、要求路徑、用戶端IP或地理位置等指標）的自訂規則來封鎖第7層攻擊。

## Web應用程式防火牆(WAF)

Ambly Web Application Firewall(WAF)用於提供額外的保護。 Apply的基於雲的WAF使用來自商業和開源源（如OWASP核心規則集）的第三方規則。 此外，還採用Adobe Commerce特定規則。 客戶受到關鍵應用程式層攻擊的保護，這些攻擊包括注入攻擊和惡意輸入、跨站點指令碼、資料過濾、HTTP協定違規以及其他OWASP前10大威脅。

如果檢測到新漏洞，Adobe Commerce將更新WAF規則，使Managed Services能夠在軟體修補程式之前「實際修補」安全問題。 Embly WAF不提供速率限制或機器人檢測服務。 如有需要，客戶可以授權與Ampley相容的協力廠商機器人偵測服務。

## Virtual Private Cloud(VPC)

Adobe Commerce Pro計畫生產環境配置為虛擬專用雲(VPC)，因此生產伺服器被隔離，並且連接到雲環境和連接到雲環境外的能力有限。 僅允許安全連接到雲伺服器。 SFTP或rsync等安全通訊協定可用於檔案傳輸。

客戶可使用SSH通道來保護與應用程式的通訊安全。 您可以支付額外費用來存取AWS PrivateLink。 所有與這些伺服器的連線都是使用AWS安全性群組來控制，這是一個虛擬防火牆，可限制與環境的連線。 客戶的技術資源可使用SSH來存取這些伺服器。

## 加密

Amazon Elastic Block Store(EBS)用於儲存。 所有EBS卷都使用AES-265算法進行加密。 這表示資料會在閒置時加密。 系統也會加密CDN與來源伺服器之間以及來源伺服器之間傳輸的資料。 客戶密碼會儲存為雜湊。 敏感憑證（包括支付閘道的憑證）會使用SHA-256演算法加密。

當資料不在休息狀態或未在伺服器之間傳輸時，Adobe Commerce應用程式不支援欄或列層級加密或加密。 客戶可以從應用程式內管理加密密鑰。 系統使用的金鑰會儲存在AWS金鑰管理系統中，且必須由Managed Services管理，才能提供部分服務。

## 滲透測試

Managed Services會透過現成可用的應用程式，定期對Adobe Commerce雲端系統進行滲透測試。 客戶負責對其自訂應用程式進行任何滲透測試。

## 支付網關

Adobe Commerce需要支付閘道整合，信用卡資料會直接從消費者的瀏覽器傳遞至支付閘道。 任何Adobe Commerce Pro計畫Managed Services環境中都無法使用卡資料。 使用來自網關的對事務的引用完成電子商務應用程式對事務的操作。

## Adobe Commerce應用程式

Adobe會定期測試核心應用程式程式碼是否有安全漏洞。 為客戶提供了缺陷和安全問題的修補程式。 產品安全團隊會依照OWASP應用程式安全性准則驗證Adobe Commerce產品。 多個安全漏洞評估工具和外部供應商用於測試和驗證合規性。 安全工具包括：

- Veracode靜態和動態掃描
- RIPS原始碼掃描
- Trustwave和Alert邏輯的漏洞掃描服務
- Burp Suite Pro
- 奧瓦斯帕普
- 和SqlMap

系統會以這些工具每兩週掃描完整的程式碼基底。 客戶會透過直接電子郵件、應用程式中的通知，以及 [安全中心](https://magento.com/security).

客戶必鬚根據PCI指南，確保在發行後30天內將這些修補程式應用到其自定義應用程式。 Adobe也提供 [安全掃描工具](https://docs.magento.com/user-guide/magento/security-scan.html) 使商家能夠定期監控其網站，並接收已知安全風險、惡意軟體和未經授權的訪問的更新。 安全掃描工具是免費服務，可在任何版本的Adobe Commerce上執行。

為了鼓勵安全研究人員識別並報告漏洞，Adobe Commerce有 [bug-bounty計畫](https://hackerone.com/magento) 除了內部測試。 此外，如果需要，向客戶提供應用程式的完整原始碼，供其自己審查。

## 只讀檔案系統

所有可執行代碼都部署在只讀檔案系統映像中，這大大減少了可用於攻擊的表面。 部署過程會建立Swach-FS映像。 此方法可大幅減少將PHP或JavaScript程式碼插入系統或修改Adobe Commerce應用程式檔案的機會。

## 遠程部署

將可執行的程式碼放入Managed Services生產環境的唯一方法，是透過布建程式執行。 這包括將原始碼從源儲存庫推送到遠程儲存庫，以啟動部署過程。 對該部署目標的訪問受控制，因此您可以完全控制誰可以訪問該部署目標。 所有應用程式程式碼部署至非生產環境的作業，均由客戶控制。

## 記錄

所有AWS活動都記錄在AWS CloudTrail中。 Linux、應用程式伺服器和資料庫日誌儲存在生產伺服器上，並儲存在備份中。 所有原始碼變更都會記錄在Git存放庫中。 可在Adobe Commerce中取得部署記錄 [項目Web介面](https://devdocs.magento.com/cloud/project/projects.html#login). 記錄所有支援存取，並記錄支援工作階段。

## 敏感資料

敏感資料可涵蓋消費者的個人資訊或Managed Services客戶的機密資料。 保護敏感客戶和消費者資料是Adobe Commerce Managed Services的重要義務。 Managed Services和我們的客戶對於個人識別資訊都有法律義務。 除了架構的安全功能外，還有其他控制項可限制敏感資料的發佈和存取。

客戶擁有其資料，並可控制該資料的位置。 客戶會指定其生產和開發執行個體所在的位置。 它們也會指定Adobe Commerce報表環境與商務搭配使用的位置，以及Adobe Commerce報表應用程式是否可存取個人識別資訊。 生產執行個體可位於大部分的AWS地區，而開發和Adobe Commerce報表環境目前可在美國或歐盟中找到。

敏感資料可能會通過Ambey CDN伺服器網路，但不會儲存在Ambeynet網路中。 Adobe Commerce Managed Services產品中包含的所有合作夥伴都有確保敏感資料保護的合約義務。 Managed Services不會將敏感客戶或消費者資料從客戶指定的位置移動。

為了提供Adobe Commerce Managed Services產品中包含的服務，Managed Services員工需要存取包含敏感資料的生產系統。 這些員工接受有關保護敏感客戶和消費者資料的義務的培訓。 對這些系統的訪問僅限於需要訪問的員工。 這些員工只能在提供這些服務所需的時間內訪問。

## 一般資料保護規範(GDPR)

GDPR是一個法律架構，為歐盟(EU)的個人設定收集和處理個人資訊的准則。 無論網站的基地為何，都適用這些法規，且歐盟訪客可存取。

基本上，訪客必須收到網站從他們收集到的資料的通知，並明確同意收集資訊。 如果網站所保存的個人資料被破壞，網站必須通知訪客。

經營網站的商家或公司也必須有專門的資料保護專員，負責監督網站的資料安全性，當訪客要求清除其資料時，該個人（或網站管理團隊）應可聯絡。

GDPR也要求收集的任何個人識別資訊（例如姓名、種族和出生日期）匿名或匿名。

## 備份

在過去24小時的操作中，每小時執行一次備份。 在24小時後，使用AWS EBS快照服務，備份將按以下時間表保留：

| 時段 | 備份保留策略 |
|----------------|-------------------------|
| 1至3天 | 每個備份 |
| 4至6天 | 每天一次備份 |
| 第2至6週 | 每週一次備份 |
| 8至12週 | 每兩週一次備份 |
| 第12至22週 | 每月一次備份 |

這將在冗餘儲存上建立獨立備份。 由於EBS卷已加密，因此備份也已加密。 此外，Managed Services還根據客戶要求執行按需備份。

您的Adobe Commerce Managed Services備份和恢複方法使用高可用性體系結構和全系統備份。 每個專案都會跨三個不同的AWS可用區複製（所有資料、程式碼和資產）;每個區域都有一個單獨的資料中心。
